# תלמידון (Talmidon) — מסמך אפיון דרישות

> מערכת לניהול ומעקב עבור מורות פרטיות.
> פונה ל: מורות, תלמידים (כל הגילים), והורים.
>
> **סטטוס:** שלב 1 (אפיון דרישות) — מאושר ✅
> **עודכן:** 2026-06-25

---

## 1. מטרת המערכת

פלטפורמה **רב-דיירת (Multi-Tenant)** שבה מורות פרטיות רבות נרשמות באופן עצמאי,
כל מורה מנהלת את התלמידים שלה בלבד, עם בידוד מידע מלא בין המורות.
מעל הכל — שכבה ציבורית פתוחה (ספריית מורות) ללא צורך בהתחברות.

---

## 2. סוגי משתמשים (Roles)

| תפקיד | התחברות | תיאור |
|-------|:-------:|-------|
| **מבקר (ציבורי)** | ❌ | דף כללי פתוח עם רשימת כל המורות, תחומיהן ופרטי יצירת קשר |
| **מורה** | ✅ | בעלת המרחב שלה — מנהלת תלמידים, יומן, תשלומים, הערות, ודף כללים אישי |
| **הורה** | ✅ | צופה במידע של הילד/ים שלו בלבד |
| **תלמיד** | ✅ | צופה במידע האישי שלו בלבד |
| *(מנהל פלטפורמה)* | ✅ | תפקיד-על מינימלי לתחזוקה (להגדרה בשלב מאוחר) |

---

## 3. מודל הרשאות מפורט

### מורה
- נרשמת **בעצמה** לפלטפורמה (הרשמה עצמאית, ללא אישור ידני).
- יוצרת ידנית את חשבונות התלמידים וההורים.
- רואה **רק** את התלמידים שלה — בידוד מלא ממורות אחרות.
- מנהלת יומן שיעורים לכל תלמיד.
- בסוף כל שיעור מסמנת: התקיים + האם נדרש תשלום עליו.
- כותבת הערות פדגוגיות, עם בקרת נראוּת (ראה סעיף 5).
- מנהלת דף כללים אישי: מחיר לשיעור, כללי ביטול/תשלום, פרטי התקשרות.
- מסמנת "שולם" עם קבלת תשלום.

### הורה
- רואה את היומן של הילד/ים הספציפי/ים שלו.
- רואה **סטטוס תשלומים מלא** + הסבר (מה שולם / מה נדרש).
- מקבל תזכורת תשלום חודשית במייל.
- רואה את כל ההערות שסומנו כגלויות לו.

### תלמיד
- רואה את היומן האישי שלו.
- רואה רק את ההערות שסומנו כגלויות לתלמיד.
- ❌ **אינו** רואה סטטוס תשלומים.

---

## 4. מודולים מרכזיים

1. **כרטיס תלמיד** — פרטים אישיים, פרטי הורה, היסטוריה.
2. **מעקב פדגוגי** — הערות התקדמות עם בקרת נראוּת (תלמיד/הורה).
3. **יומן / לוח זמנים** — מתי כל תלמיד מגיע.
4. **תשלומים** — חיוב לפי שיעור, סימון ידני בסוף שיעור, מעקב שולם/נדרש, תזכורת חודשית.
5. **דף כללים אישי למורה** — מחיר לשיעור, כללים, פרטי יצירת קשר.
6. **דף ציבורי (ספרייה)** — כל המורות, תחומים ופרטי יצירת קשר. ללא כניסה.

---

## 5. כלל נראוּת ההערות

לכל הערה שהמורה כותבת יש **שני מתגים**: "גלוי לתלמיד" / "גלוי להורה".

- **ברירת מחדל:** מה שגלוי לתלמיד — גלוי אוטומטית גם להורה.
- המורה יכולה לשנות זאת ידנית (למשל הערה רגישה שתישלח רק להורה, לא לתלמיד).

---

## 6. זרימת התשלום (Payment Flow)

1. בסוף שיעור — המורה מסמנת אם נדרש תשלום.
2. **חיוב לפי שיעור** (לא מנוי חודשי).
3. פעם בחודש — תזכורת תשלום **במייל** להורה (מרכזת את החיובים הפתוחים).
4. כשמתקבל תשלום — המורה מסמנת "שולם" → המערכת שולחת **מייל אישור תשלום** להורה.

> **הערה:** בשלב זה אין סליקת אשראי אונליין — מעקב ידני בלבד.

---

## 7. ערוץ התקשרות + מטריצת התראות מייל

**מייל (אימייל)** הוא ערוץ ההתראות היחיד. נדרש שירות שליחת מיילים.

| אירוע | מי מבצע | נמען המייל |
|-------|---------|-----------|
| תזכורת תשלום חודשית | מערכת (אוטומטי, פעם בחודש) | הורה |
| אישור קבלת תשלום | מורה מסמנת "שולם" | הורה |
| הוספת שיעור | מורה | הורה |
| עדכון שיעור (מועד/פרטים) | מורה | הורה |
| ביטול/מחיקת שיעור | מורה | הורה |
| בקשה לקביעת שיעור | הורה | מורה |
| בקשה לביטול שיעור | הורה | מורה |
| בקשה לעדכון שיעור | הורה | מורה |

> כלל: שינויים שהמורה עושה ביומן → התראה **להורה**.
> בקשות שההורה שולח (קביעה/ביטול/עדכון) → התראה **למורה**.

## 8. סטאק טכנולוגי (שלב 2 — מאושר ✅)

| שכבה | טכנולוגיה |
|------|-----------|
| **Frontend** | Angular (TypeScript) + **PrimeNG** (נושא Aura) + PrimeFlex + PrimeIcons, עברית RTL |
| **Backend** | ASP.NET Core Web API (C#) |
| **ORM** | Entity Framework Core |
| **בסיס נתונים** | PostgreSQL (ספק Npgsql) |
| **התחברות/הרשאות** | ASP.NET Core Identity + JWT, 4 תפקידים |
| **בידוד רב-דיירת** | EF Core Global Query Filters לפי `TenantId` |
| **שליחת מיילים** | שירות SMTP/API (SendGrid / Mailgun / Azure — להחלטה בשלב 5) |
| **תזכורת חודשית** | Background Job (Hangfire / Quartz.NET) |

**עיקרון בידוד מרכזי:** לכל ישות שייכת למורה יוצמד `TenantId`, ו-Global Query Filter
ב-EF Core יבטיח שמורה לעולם לא תשלוף נתונים של מורה אחרת — גם ללא סינון ידני.
